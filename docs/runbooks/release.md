# Runbook: Releasing a gnunix version

The release pipeline is the operator-facing half of [ADR-008](../adrs/ADR-008-renovate-and-release.md).
Three workflows participate, each with a single responsibility:

| Workflow | File | Trigger | What it does |
|---|---|---|---|
| Tag on version bump | `.github/workflows/tag-on-version-bump.yml` | push to `main` touching `tools/manifest.json` | If `lfs_image_version` changed, create + push `v<version>` tag |
| Build | `.github/workflows/build.yml` | push tag `v*` / dispatch | Fetch each image from `gnunix-images-<ver>` and re-upload as a workflow artifact for `release.yml` to attach (per PR-3b.1; rebuild path returns in PR-3b.2 — issue #20) |
| Release | `.github/workflows/release.yml` | push tag `v*` / dispatch | Wait for Build, download artifacts, draft a GH Release |

## The happy path

```text
  ┌─────────────────────────┐
  │ PR bumps                │
  │ tools/manifest.json :   │   merge to main
  │  lfs_image_version      ├──────────────────┐
  └─────────────────────────┘                  │
                                               ▼
                              ┌─────────────────────────────────┐
                              │ tag-on-version-bump.yml         │
                              │  detects bump,                  │
                              │  pushes annotated tag vX.Y.Z    │
                              └────────────────┬────────────────┘
                                               │
                              ┌────────────────┴────────────────┐
                              ▼                                 ▼
                ┌─────────────────────────────┐     ┌──────────────────────────┐
                │ build.yml @ vX.Y.Z          │     │ release.yml @ vX.Y.Z     │
                │  fetch-base                 │     │  polls for build.yml run │
                │  fetch-minimal              │     │   on same SHA            │
                │  fetch-desktop              │     │  downloads artifacts     │
                │  fetch-installer (soft)     │     │  creates DRAFT release   │
                │ ← gnunix-images-<ver> release │     │                          │
                └─────────────────────────────┘     └──────────────────────────┘
                              │                                 ▲
                              └─────────────────────────────────┘
                                  upload-artifact   gh run download
```

End state: a **draft** release on the GitHub Releases tab. A human reviews
the asset list and clicks **Publish**.

## Prerequisite — publish to `gnunix-images-<ver>` first

Until PR-3b.2 lands the qemu+KVM rebuild path (tracking issue #20), `build.yml`
**does not rebuild** any image. It fetches them from the intermediate
`gnunix-images-<ver>` release. The maintainer must populate that release from
a Mac before pushing the `v<ver>` tag:

```sh
# On the maintainer's Mac, after a successful local rebuild:
tools/build-all.sh gnunix-base       # ~6–10h (skip if pins didn't change)
tools/build-all.sh gnunix-minimal
tools/build-all.sh gnunix-desktop
tools/build-all.sh gnunix-installer  # optional; soft-fails in CI if absent

# Publish all four to the SAME tag (note --release-tag override for
# desktop/installer, whose default would otherwise be v<ver>):
V=$(jq -r .lfs_image_version tools/manifest.json)
tools/release-image.sh gnunix-base
tools/release-image.sh gnunix-minimal
tools/release-image.sh gnunix-desktop   --release-tag=gnunix-images-$V
tools/release-image.sh gnunix-installer --release-tag=gnunix-images-$V

# Now push the user-facing tag — the rest of the pipeline runs in CI.
git push origin v$V
```

When PR-3b.2 lands, the desktop/installer rows above become unnecessary —
CI will rebuild them on top of fetched minimal. The base/minimal rows stay,
since ADR-021 keeps base off the hosted-runner path forever.

## What ships per release

Per the [platform matrix](platforms.md):

### Base images (always present in a valid release)

- `gnunix-base-disk-<ver>.img.zst`
- `gnunix-minimal-disk-<ver>.img.zst`
- `gnunix-desktop-disk-<ver>.img.zst`

`release.yml` refuses to draft a release if any of these are missing.

### Platform-packaged (present when the platform is shipping)

| Platform | Arch | Pattern | Status today |
|---|---|---|---|
| `generic-uefi` | aarch64 | `gnunix-{minimal,desktop}-generic-uefi-aarch64-<ver>.img.zst` | shipping |
| `rpi-native` | aarch64 | `gnunix-{minimal,desktop}-rpi-native-aarch64-<ver>.img.zst` | scaffolded |
| `nuc-installer` | x86_64 | `gnunix-{minimal,desktop}-nuc-installer-x86_64-<ver>.iso` | scaffolded |

Scaffolded platforms are listed in the release body but the file column
shows `—` until the platform's prerequisites in `docs/TODO.md` land.

### Always attached

- `SHA256SUMS` — generated by `release.yml` over every artifact actually
  uploaded.
- `manifest.json` — pinned versions for reproducibility.

## Triggering a release

### Standard flow — bump and merge

```sh
# 1. Bump the version on a branch.
jq '.lfs_image_version = "0.2.0"' tools/manifest.json > tmp && mv tmp tools/manifest.json
git checkout -b bump-v0.2.0
git commit -am "release: 0.2.0"
gh pr create --fill

# 2. Merge to main. tag-on-version-bump.yml creates v0.2.0 automatically.
# 3. build.yml runs on v0.2.0; release.yml waits for it and drafts a release.
# 4. Review the draft at https://github.com/<owner>/<repo>/releases and click Publish.
```

### Manual tag (skips the auto-tag step)

```sh
git tag -a v0.2.0 -m "gnunix 0.2.0"
git push origin v0.2.0
# build.yml + release.yml run as above.
```

### Re-publish an existing tag (e.g., fixed release notes)

```sh
gh workflow run release.yml -f tag=v0.2.0
# release.yml deletes the existing draft and re-creates it from the same
# build run.
```

## Tolerated failure modes

`release.yml` is conservative about what it lets through:

| Situation | release.yml behaviour |
|---|---|
| `build.yml` run succeeded for all jobs | Drafts the release |
| `gnunix-desktop` job failed but base + nix succeeded | Refuses to publish (base sanity check fails on missing `gnunix-desktop-disk-*.img.zst`) |
| `package` matrix slot for scaffolded platform exited `rc=2`/`rc=3` | Drafts the release; the scaffolded row shows `—` in the table |
| `build.yml` hasn't started yet on the tag SHA | Polls for up to 8h, then fails the release workflow |
| Tag value disagrees with `tools/manifest.json:lfs_image_version` | Fails before downloading anything |

The 8h poll-for-build-run timeout in `release.yml` was originally sized for
the worst-case `gnunix-base` build on M-series hardware. After PR-3b.1 the
fetch jobs finish in minutes; the long timeout is intentionally kept so the
same workflow accommodates PR-3b.2's qemu rebuild jobs without further
tuning.

## Untrigger / abort

If you want to abort a release that's mid-flight:

```sh
# Cancel the in-flight Release workflow run.
gh run cancel <run-id>

# Delete the tag (only if no Release has been published yet).
git push --delete origin v0.2.0
git tag -d v0.2.0

# If a draft release was already created, delete it.
gh release delete v0.2.0 --yes
```

If a Release has been *published* (non-draft), don't delete it; cut a new
patch version (`v0.2.1`) that supersedes it.

## Manifest <-> tag consistency

The hard invariant: `git describe --tags --exact-match` on the release SHA
equals `v$(jq -r .lfs_image_version tools/manifest.json)`. Both
`tag-on-version-bump.yml` and `release.yml` enforce this. If you've
violated it (e.g., hand-pushed a `v0.2.0` tag while the manifest still says
`0.1.0`), `release.yml` fails the meta-resolve step with a clear error.

## Why this shape

- **Manifest is the single source of truth for the version.** Renovate
  already bumps `tools/manifest.json`; the auto-tagger just turns that bump
  into the tag we'd otherwise type by hand.
- **Build and release are separate workflows.** Build assembles the
  artifact set on a hosted runner — today by fetching from
  `gnunix-images-<ver>` (PR-3b.1); in PR-3b.2 by fetching base+minimal and
  rebuilding desktop+installer via qemu+KVM. Release is light
  (`gh release create`, seconds) and can be re-run without rebuilding.
- **Draft, not auto-publish.** Single-maintainer audience (ADR-005); one
  click is cheaper than rolling back a broken auto-publish.
- **Platform matrix awareness.** `release.yml` describes every row of the
  matrix but only attaches what was actually built. Scaffolded rows show
  `—` until they're real.
